## 1. УЛУЧШЕНИЕ АЛГОРИТМА ОБНАРУЖЕНИЯ ORDER BLOCKS (OB)

### 1.1. Модернизация кластерного алгоритма `f_cluster()`

#### 1.1.1. Проблема: Хрупкость кластерного обнаружения
**Текущее поведение**: Сканирование назад от пивота (`swing_idx`) ищет последовательные свечи одного направления (красные для бычьих OB, зелёные для медвежьих). Одна «шумовая» свеча противоположного цвета разрывает кластер.

**План улучшения**:
- 1.1.1.a. Ввести параметр `noise_tolerance` (0-2, default 1) — количество допустимых «шумовых» свечей внутри кластера
- 1.1.1.b. Реализовать алгоритм «мягкого кластера»: вместо жёсткого `if candle_color != expected → stop`, использовать скользящее окно с порогом доминирования (>70% свечей кластера должны быть нужного цвета)
- 1.1.1.c. Добавить весовую систему внутри кластера: свечи с большим объёмом и z-score получают больший вес при определении границ зоны
- 1.1.1.d. Реализовать двухпроходный алгоритм: первый проход — грубое обнаружение кандидата, второй — уточнение границ с учётом объёмного профиля

#### 1.1.2. Проблема: Статические границы кластера
**Текущее поведение**: Границы зоны = max(high) и min(low) свечей кластера, с допуском `p_cluster_tol × ATR`.

**План улучшения**:
- 1.1.2.a. Реализовать объёмно-взвешенные границы: `zone_top = VWAP(high, volume)` по свечам кластера, `zone_bot = VWAP(low, volume)` — смещение к ценовым уровням с максимальным объёмом
- 1.1.2.b. Добавить адаптивный допуск кластера: `adaptive_tol = cluster_tol × (1 + vol_ratio/3)` — при высоком объёме расширять допуск для захвата более значимых зон
- 1.1.2.c. Реализовать «ядро зоны» (core zone) — подзона с максимальной концентрацией объёма (аналог Value Area в Volume Profile), и «буферную зону» (buffer zone) — полная ширина кластера

#### 1.1.3. Проблема: Отсутствие мульти-таймфреймной кластеризации
**Текущее поведение**: Кластеры ищутся только на текущем таймфрейме.

**План улучшения**:
- 1.1.3.a. Реализовать функцию `f_htf_cluster_confirmation()`: запрос `request.security()` для HTF данных и проверка совпадения зоны текущего TF с зоной HTF (допуск 1.5×ATR)
- 1.1.3.b. При подтверждении HTF — бонус +25 к скору зоны, при отсутствии — нет штрафа (не блокировать, но понижать приоритет)
- 1.1.3.c. Передавать HTF-подтверждение через новый канал `ENG: HTF Zone Confirm` (кодировать как 0/1 для экономии plot-слотов)

### 1.2. Улучшение системы валидации OB-зон

#### 1.2.1. Многоуровневая валидация свечного паттерна
**Текущее поведение**: Простая проверка wick ratio (`inp_wick_rat`) и shadow filter (`lower_shadow_min`, `upper_shadow_min`).

**План улучшения**:
- 1.2.1.a. Добавить **Volume Spread Analysis (VSA)** при валидации: проверять наличие No Demand / No Supply / Stopping Volume в зоне кластера — зоны с Stopping Volume получают +15 к скору
- 1.2.1.b. Добавить **Wyckoff Spring/Upthrust** детекцию прямо в кластерный алгоритм: если последняя свеча кластера имеет признаки Spring (для бычьих OB) или Upthrust (для медвежьих), то бонус +20
- 1.2.1.c. Реализовать **Absorption Detection** на уровне кластера: если внутри кластера есть свеча с volume > 2.0× среднего И body < 0.3× среднего, это сигнал поглощения — бонус +12
- 1.2.1.d. Добавить **Order Flow Imbalance** детекцию: измерять delta (объём покупок vs продаж) через формулу `volume × (2*(close-low)/(high-low) - 1)` для каждой свечи кластера — положительная суммарная дельта для бычьего OB = +10

#### 1.2.2. Контекстная валидация
**Текущее поведение**: Валидация привязана только к свойствам самой свечи.

**План улучшения**:
- 1.2.2.a. Добавить **структурную валидацию**: OB должен находиться на структурном уровне (HH/HL для бычьего, LH/LL для медвежьего) — если зона не на структурном уровне, штраф -15
- 1.2.2.b. Реализовать **ликвидность-осведомлённую** валидацию: проверить наличие равных максимумов/минимумов (EQH/EQL) вблизи зоны (допуск 0.3×ATR) — наличие ликвидности = бонус +10
- 1.2.2.c. Добавить **импульсную валидацию**: движение от зоны должно быть «импульсным» (≥ 2 свечи одного направления с body > 60% range) — отсутствие импульса после зоны = штраф -10
- 1.2.2.d. Реализовать **институциональную валидацию**: проверка кластера крупных объёмных свечей (RVOL > 2.5) на подходе к зоне — признак институционального интереса = бонус +15

### 1.3. Улучшение жизненного цикла OB-зон

#### 1.3.1. Адаптивное старение зон
**Текущее поведение**: `decay_factor = exp(-0.001 × age_bars)` — фиксированная скорость затухания для всех зон.

**План улучшения**:
- 1.3.1.a. Реализовать **адаптивный decay**: скорость затухания зависит от начального скора зоны. Высококачественные зоны (score > 100) затухают медленнее: `decay_rate = 0.001 × (150 / max(score, 50))`
- 1.3.1.b. Добавить **ретест-основанное обновление возраста**: при каждом ретесте зоны (цена касается, но не пробивает) сбрасывать часть возраста: `age := age × 0.7` — зоны, которые регулярно тестируются, остаются «свежими»
- 1.3.1.c. Реализовать **сессионное обновление возраста**: зоны, созданные во время kill zone (London Open / NY Open), получают +30% к времени жизни
- 1.3.1.d. Добавить минимальный порог decay: `decay_factor = max(0.05, exp(...))` для предотвращения полного обнуления скора

#### 1.3.2. Умная инвалидация зон
**Текущее поведение**: Зона инвалидируется при пробое (close за границей зоны) или при превышении дистанции `inp_inv_dist × ATR`.

**План улучшения**:
- 1.3.2.a. Реализовать **градуированную инвалидацию**: вместо бинарного «активна/неактивна», ввести 3 состояния: Active (100%), Weakened (50%, после 1 пробоя), Invalidated (0%, после 2 пробоев или close + retest за границей)
- 1.3.2.b. Добавить **объёмную инвалидацию**: пробой с RVOL > 3.0 = немедленная инвалидация (без перехода в Weakened)
- 1.3.2.c. Реализовать **временную инвалидацию**: если цена проводит > 5 баров за границей зоны с volume выше среднего, зона инвалидируется даже без чистого пробоя
- 1.3.2.d. Добавить **контекстную защиту от ложных пробоев**: если пробой происходит на малом объёме (RVOL < 0.8) и цена возвращается в зону в течение 3 баров → это sweep, а не инвалидация → бонус +8

## 2. УЛУЧШЕНИЕ АЛГОРИТМА REJECTION ZONES (RZ)

### 2.1. Модернизация детекции wick rejection

#### 2.1.1. Проблема: Зависимость только от wick-to-body ratio
**Текущее поведение**: Bull RZ = lower wick ≥ `wick_mult × body` И ≥ `0.3 × ATR`. Это пропускает высококачественные отклонения с нестандартной морфологией.

**План улучшения**:
- 2.1.1.a. Добавить **мульти-бар rejection** детекцию: не только одна свеча с длинной тенью, но и паттерн 2-3 свечей, формирующих отклонение (например, pin bar + inside bar + подтверждающая свеча)
- 2.1.1.b. Реализовать **кумулятивный wick** анализ: суммировать нижние тени 3-5 свечей в зоне — если суммарная длина > 4×ATR, это мощная зона отклонения даже без одиночного пин-бара
- 2.1.1.c. Добавить **Volume-Weighted Rejection Score**: wick_score = Σ(wick_length × volume_weight) для каждой свечи в зоне — учитывает как размер тени, так и объём
- 2.1.1.d. Реализовать **дельта-rejection**: если суммарная дельта (buy_vol - sell_vol) в нижней зоне тени > 0 для бычьего RZ, это подтверждение агрессивных покупок на уровне — бонус +15

#### 2.1.2. Проблема: Sniper price = midpoint зоны — неоптимально
**Текущее поведение**: `sniper_price = (top + bot) / 2` для RZ-зон.

**План улучшения**:
- 2.1.2.a. Реализовать **wick-weighted sniper**: `sniper_price = top - wick_depth × sniper_ratio`, где `wick_depth = точка максимального отклонения тени`, `sniper_ratio = 0.618` (Fibonacci) — точка входа ближе к точке реального отклонения
- 2.1.2.b. Добавить **Volume-POC sniper**: рассчитывать Point of Control (уровень максимального объёма) внутри зоны RZ и использовать его как sniper price
- 2.1.2.c. Реализовать **адаптивный sniper на основе волатильности**: при высокой волатильности (ATR > 1.5× среднего) sniper сдвигается глубже в зону (0.618), при низкой — ближе к краю (0.382)

### 2.2. Улучшение RSI Divergence для RZ

#### 2.2.1. Расширение дивергенции
**Текущее поведение**: Только классическая дивергенция (price lower low + RSI higher low для bull).

**План улучшения**:
- 2.2.1.a. Добавить **скрытую дивергенцию**: price higher low + RSI lower low = продолжение тренда → бонус +10 для RZ в направлении тренда
- 2.2.1.b. Реализовать **мульти-индикаторную дивергенцию**: проверять дивергенцию не только RSI, но и Stochastic, MACD, OBV — каждая дополнительная дивергенция = +5 к скору (max +15)
- 2.2.1.c. Добавить **таймфрейм-подтверждённую дивергенцию**: если дивергенция подтверждается на HTF → бонус +20 (мощный сигнал разворота)

### 2.3. Улучшение Quadratic Wick Bonus

#### 2.3.1. Проблема: Неограниченный бонус
**Текущее поведение**: `pow(min(wick_ratio, 3), 2) × 20` → при wick_ratio = 3.0 bonus = 180, что превышает cap 150.

**План улучшения**:
- 2.3.1.a. Заменить квадратичную функцию на **сигмоидную**: `wick_bonus = 30 / (1 + exp(-3 × (wick_ratio - 2)))` — плавное насыщение на уровне 30 при wick_ratio > 3
- 2.3.1.b. Добавить **контекстный множитель**: wick_bonus × (1 + 0.3 × is_kill_zone + 0.2 × is_high_vol) — усиленные отклонения в kill zone и при высоком объёме
- 2.3.1.c. Жёсткий cap: `wick_bonus = min(wick_bonus, 35)` — гарантия, что wick_bonus не доминирует в итоговом скоре

## 3. УЛУЧШЕНИЕ FAIR VALUE GAP (FVG) СИСТЕМЫ

### 3.1. Расширение типов FVG

#### 3.1.1. Текущие типы FVG — 4 варианта
**Текущее поведение**: Classic FVG, Vol Imbalance, Marubozu, Open-Open Imbalance.

**План улучшения**:
- 3.1.1.a. Добавить **Liquidity Void FVG**: серия из 3+ свечей одного направления с минимальным перекрытием теней (< 20% среднего range) — представляет область полного отсутствия ликвидности
- 3.1.1.b. Реализовать **Balanced Price Range (BPR) Score**: текущая детекция BPR = бинарная (есть/нет). Улучшить до градуированной: `bpr_score = overlap_pct × 20` — больше перекрытие = сильнее сигнал
- 3.1.1.c. Добавить **Opening Range FVG**: gap между закрытием предыдущей сессии и открытием текущей (для фьючерсных пар, особенно на OKX) — специальный бонус +12
- 3.1.1.d. Реализовать **Micro FVG**: FVG на тиковых данных (используя close, open вместо high/low) для таймфреймов M1-M3 — повышенная точность для скальпинга

### 3.2. Улучшение FVG-заполнения и отслеживания

#### 3.2.1. Проблема: FVG fill penalty слишком узкий
**Текущее поведение**: При `fvg_fill_pct > 0.5`: `score -= 10.0 × z.fvg_fill_pct` — диапазон штрафа 5.0-8.0.

**План улучшения**:
- 3.2.1.a. Реализовать **нелинейный штраф**: `penalty = 25 × (fill_pct - 0.5)² / 0.25` — квадратичный рост от 0 при 50% до 25 при 100% заполнения
- 3.2.1.b. Добавить **объёмно-взвешенное заполнение**: вместо чисто ценового fill_pct, учитывать объём сделок в заполненной области: `vol_fill_pct = vol_in_filled / vol_total` — если заполнение произошло на малом объёме, штраф уменьшается на 40%
- 3.2.1.c. Реализовать **временное заполнение**: если FVG заполняется в первые 5 баров после создания, это слабый FVG (немедленный штраф -15), если заполнение начинается после 20+ баров — FVG «закалённый» (штраф уменьшен на 50%)

### 3.3. FVG и зоны — интеграция

#### 3.3.1. Проблема: FVG sequence lock слишком агрессивный
**Текущее поведение**: `active_bull_fvg_lock` блокирует создание новых бычьих зон если цена внутри любого активного бычьего FVG.

**План улучшения**:
- 3.3.1.a. Ограничить lock только FVG моложе 50 баров И с fill_pct < 40% — старые/частично заполненные FVG не должны блокировать новые зоны
- 3.3.1.b. Реализовать **FVG Priority Queue**: ранжировать активные FVG по `(1 - fill_pct) × volume × decay_factor` — lock активируется только для top-3 FVG
- 3.3.1.c. Добавить **FVG-Zone Synergy Score**: если зона OB/RZ пересекается с активным FVG → суммарный бонус +15 (вместо текущего Propulsion Block бонуса +20, который работает только для same-candle совпадения)

## 4. УЛУЧШЕНИЕ BREAKER BLOCKS (BB)

### 4.1. Модернизация конвертации OB → BB

#### 4.1.1. Проблема: Пересчёт скора при BB слишком прост
**Текущее поведение**: `new_score = 0.8 × old_score + momentum + bonuses`.

**План улучшения**:
- 4.1.1.a. Реализовать **контекстный пересчёт**: `new_score = old_score × break_quality_multiplier`, где `break_quality_multiplier = f(break_volume, break_speed, break_distance)`:
  - `break_volume` = RVOL на свече пробоя (> 2.0 = 1.2×, > 3.0 = 1.4×)
  - `break_speed` = количество свечей от входа в зону до пробоя (1-2 свечи = aggressive = 1.3×, > 5 = slow = 0.8×)
  - `break_distance` = насколько далеко цена ушла за зону / ATR (> 2.0 ATR = 1.2×, < 0.5 ATR = 0.7×)
- 4.1.1.b. Добавить **импульс после пробоя**: если после конвертации в BB следует серия из 3+ импульсных свечей в направлении пробоя → бонус +15
- 4.1.1.c. Реализовать **ретест-подтверждённый BB**: если цена возвращается к BB и отклоняется (wick rejection) → бонус +20, это подтверждает смену полярности

#### 4.1.2. Проблема: Cascade limit = 2 — фиксированный
**Текущее поведение**: После 2 пробоев BB деактивируется.

**План улучшения**:
- 4.1.2.a. Реализовать **адаптивный cascade limit**: `max_breaks = 2 + floor(original_score / 50)` — высококачественные зоны (score > 100) допускают до 4 пробоев
- 4.1.2.b. Добавить **cascade scoring**: каждый последующий пробой уменьшает score на 30% (вместо полной деактивации после N пробоев)

### 4.2. Новый тип: Mitigation Block Enhancement

#### 4.2.1. Расширение Mitigation логики
**Текущее поведение**: BB + ретест + удержание = +20 бонус.

**План улучшения**:
- 4.2.1.a. Реализовать **multi-retest scoring**: первый ретест = +20, второй = +10, третий = +5 (убывающий бонус за каждый успешный ретест)
- 4.2.1.b. Добавить **качество ретеста**: оценивать глубину проникновения при ретесте — shallow (< 30% зоны) = +5, medium (30-60%) = +15, deep (60-90%) = +25
- 4.2.1.c. Реализовать **объёмный ретест**: если ретест сопровождается высоким объёмом (RVOL > 2.0), множитель ×1.5 к бонусу

## 5. УЛУЧШЕНИЕ CONFLUENCE SCORING SYSTEM

### 5.1. Расширение компонентной модели

#### 5.1.1. Текущие 6 компонент → расширить до 9
**Текущие**: Z-Score (25%), Volume/RVOL (15%), Trend Alignment (15%), Sweep/Liquidity (15%), Efficiency/FER (15%), PA Bonus (15%).

**План улучшения**:
- 5.1.1.a. Добавить **Market Regime Alignment** (новый компонент, вес 10%): зона в тренде при trending regime = полный вес, зона против тренда при trending = 0, зона в ranging = 0.5 (нейтрально)
- 5.1.1.b. Добавить **Session Quality** (новый компонент, вес 8%): зоны, созданные в London/NY kill zone = полный вес, Asia = 0.5, quiet = 0.2
- 5.1.1.c. Добавить **Multi-TF Confluence** (новый компонент, вес 7%): количество таймфреймов, на которых зона подтверждена (1 TF = 0.3, 2 TF = 0.7, 3+ TF = 1.0)
- 5.1.1.d. Перебалансировать веса: Z-Score (20%), Volume (12%), Trend (12%), Sweep (12%), Efficiency (10%), PA (10%), Regime (10%), Session (7%), Multi-TF (7%)

#### 5.1.2. Проблема: Trend Alignment — бинарный
**Текущее поведение**: 0 или full weight, без градации.

**План улучшения**:
- 5.1.2.a. Реализовать **3-уровневую градацию**: Full alignment (HTF + ITF совпадают) = 1.0, Partial (ITF совпадает, HTF нет) = 0.5, Counter-trend (оба против) = 0.0
- 5.1.2.b. Добавить **динамическую оценку силы тренда**: weight = base_weight × min(1.0, ADX / 25) — чем сильнее тренд (по ADX), тем больше вес alignment

#### 5.1.3. Проблема: Sweep — 2 дискретных значения (0.6× или 1.0×)
**Текущее поведение**: Бинарный sweep (есть ликвидация или нет).

**План улучшения**:
- 5.1.3.a. Реализовать **непрерывную функцию sweep quality**: `sweep_score = 0.3 + 0.7 × min(1.0, sweep_depth / (1.5 × ATR))` — от 0.3 (минимальный sweep) до 1.0 (глубокий sweep > 1.5 ATR)
- 5.1.3.b. Добавить **множественный sweep бонус**: если за последние 30 баров было 2+ sweep на одном уровне → extra +0.2 к sweep_score
- 5.1.3.c. Реализовать **volume-confirmed sweep**: sweep с RVOL > 2.5 = 1.3× множитель к sweep_score

### 5.2. Нормализация и бонусная система

#### 5.2.1. Проблема: Бонусы — магические числа, аддитивные без контроля
**Текущее поведение**: 20+ различных бонусов (от +5 до +150), разбросанных по 1000+ строкам.

**План улучшения**:
- 5.2.1.a. **Собрать все бонусы в единый блок констант** (раздел после inputs) с документацией для каждого:
  ```
  BONUS_THREE_SOLDIERS = 10.0    // Три последовательных бычьих/медвежьих свечи
  BONUS_TWEEZER = 8.0            // Tweezer top/bottom паттерн
  BONUS_MARUBOZU = 12.0          // Свеча без теней (сильный импульс)
  ...
  ```
- 5.2.1.b. Реализовать **категоризацию бонусов** с лимитами по категориям:
  - Candlestick Patterns: max +25
  - Volume/Liquidity: max +20
  - Structure (BOS/CHoCH): max +20
  - Session/Timing: max +15
  - Technical (RSI/ATR): max +15
  - **Суммарный cap бонусов**: max +70 (из 150 максимального скора)
- 5.2.1.c. Реализовать **мультипликативную модель** вместо аддитивной: `final_score = base_score × (1 + category_bonus_pct)`, где category_bonus_pct — нормализованный бонус 0-100%

#### 5.2.2. Проблема: Multi-pattern scoring — ступенчатый
**Текущее поведение**: 1 паттерн = +0%, 2 = +10%, 3 = +30%, 4 = +40%, 5+ = +50%.

**План улучшения**:
- 5.2.2.a. Заменить на **логарифмическую функцию**: `multi_bonus = 15 × ln(1 + pattern_count) / ln(6)` — плавное нарастание от 0% до ~46%
- 5.2.2.b. Добавить **качество паттернов**: не все паттерны равны — weighted multi-pattern: `w_multi = Σ(pattern_quality × pattern_weight) / Σ(pattern_weight)`

### 5.3. Нейросеть — улучшение

#### 5.3.1. Проблема: 6→4→1 слишком простая
**Текущее поведение**: Простая feedforward сеть с online learning (Widrow-Hoff, lr=0.01).

**План улучшения**:
- 5.3.1.a. Расширить архитектуру до **9→8→4→1**: 9 входов (6 текущих + regime + session + hurst), 8 нейронов скрытого слоя 1, 4 скрытого слоя 2
- 5.3.1.b. Добавить **L2 регуляризацию**: `loss += lambda × Σ(w²)`, где lambda = 0.001 — предотвращение переобучения
- 5.3.1.c. Реализовать **running normalization** входов: вместо ручного `z_score/5`, `min(rvol,5)/5` → адаптивный z-score через running mean/std с окном 500 баров
- 5.3.1.d. Добавить **adaptive learning rate**: `lr = 0.01 × (1 / (1 + 0.001 × update_count))` — уменьшение lr со временем для стабилизации весов
- 5.3.1.e. Реализовать **dropout** (вероятностное обнуление 20% нейронов) через детерминистический паттерн (Pine не имеет RNG): `mask = bit(bar_index, neuron_idx)` — псевдо-случайное маскирование

## 6. УЛУЧШЕНИЕ СИСТЕМЫ PREMIUM/DISCOUNT И ЛИКВИДНОСТИ

### 6.1. Premium/Discount Zone Analysis

#### 6.1.1. Проблема: Простое сравнение close vs midpoint
**Текущее поведение**: `pd_ok = close < midpoint` для bull entry.

**План улучшения**:
- 6.1.1.a. Реализовать **многоуровневый P/D**: 
  - Premium Zone (> 75% range) — только sell entries
  - Equilibrium (40-60% range) — нейтральная, разрешены обе стороны
  - Discount Zone (< 25% range) — только buy entries
  - Deep Discount (< 10% range) — buy + бонус +10
- 6.1.1.b. Добавить **динамический диапазон P/D**: вместо fixed high/low за N баров, использовать ATR-адаптивный range: `pd_high = highest(high, max(50, floor(200 / atr_ratio)))` — при высокой волатильности укорачивать окно
- 6.1.1.c. Реализовать **Point of Control в P/D**: использовать POC Price из Volume Profile как альтернативный midpoint — более точное определение «справедливой цены»

### 6.2. Liquidity Detection Enhancement

#### 6.2.1. Расширение Sweep For Profit (SFP) алгоритма
**Текущее поведение**: Детекция sweep через `inp_sfp_lookback` (30) и `inp_sweep_ratio` (0.2).

**План улучшения**:
- 6.2.1.a. Реализовать **Liquidity Pool Map**: отслеживать кластеры стоп-лоссов (EQH/EQL за последние 100 баров) и визуализировать их как «пулы ликвидности»
- 6.2.1.b. Добавить **Inducement Detection Enhancement**: не просто проверять IDM swept, но оценивать качество inducement — глубина sweep, объём, скорость возврата
- 6.2.1.c. Реализовать **стоп-хант паттерн**: если цена пробивает EQH/EQL → немедленно разворачивается → это classic stop hunt → бонус +25 к ближайшей зоне в направлении разворота
- 6.2.1.d. Добавить **ликвидность-осведомлённый SL**: размещение SL за ближайшим пулом ликвидности, а не просто zone boundary ± ATR×pad

## 7. НОВЫЕ ТИПЫ ЗОН (РАСШИРЕНИЕ)

### 7.1. Imbalance Zones (новый тип)

#### 7.1.1. Определение и детекция
**Новый тип зоны**: область с экстремальным дисбалансом объёмов покупки/продажи.

**План реализации**:
- 7.1.1.a. Рассчитывать `delta_cumulative` на скользящем окне 5-10 баров: `delta = Σ(volume × (2*(close-low)/(high-low) - 1))`
- 7.1.1.b. Детектировать экстремальные дельта-значения: `abs(delta) > 2.0 × stdev(delta, 50)` → создать Imbalance Zone
- 7.1.1.c. Скор зоны = f(abs(delta), volume_concentration, direction_alignment)
- 7.1.1.d. Визуализация: отдельный цвет (yellow/gold) с пометкой "IMB"

### 7.2. Accumulation/Distribution Zones (новый тип)

#### 7.2.1. Wyckoff-based зоны
**Новый тип**: длительные области аккумуляции (Wyckoff Phase B) или дистрибуции.

**План реализации**:
- 7.2.1.a. Детектировать **Trading Range**: area где цена находится > 20 баров с range < 2×ATR — это потенциальная фаза аккумуляции/дистрибуции
- 7.2.1.b. Классифицировать фазу: аккумуляция = range + CVD positive trend + decreasing ATR, дистрибуция = range + CVD negative trend + decreasing ATR
- 7.2.1.c. Spring/UTAD детекция: false breakout из range → подтверждение фазы → бонус +30
- 7.2.1.d. Границы зоны = high/low trading range с буфером 0.5×ATR

### 7.3. Volatility Compression Zones (новый тип)

#### 7.3.1. VCP-based зоны
**Текущее поведение**: VCP — только бонус к существующим зонам (+8).

**План улучшения**:
- 7.3.1.a. Выделить VCP в **самостоятельный тип зоны**: серия сужающихся range (`range10 < 0.7×range20 < 0.8×range30`) с убывающим объёмом
- 7.3.1.b. Скор = f(compression_ratio, volume_decline, duration, keltner_squeeze)
- 7.3.1.c. Снайпер price = верхняя граница для бычьего breakout, нижняя для медвежьего
- 7.3.1.d. Предупреждение: VCP зоны не имеют направления до breakout — отображать как нейтральные (серые) до подтверждения

## 8. УЛУЧШЕНИЕ СТОХАСТИЧЕСКОЙ МОДЕЛИ И ПРОГНОЗИРОВАНИЯ

### 8.1. Адаптация к крипто-рынку

#### 8.1.1. Проблема: Seasonality adjustment не подходит для крипто
**Текущее поведение**: `1 + 0.1×sin(2π×hour/24) + 0.05×sin(2π×day/5)` — предполагает 24-часовой цикл и 5-дневную рабочую неделю.

**План улучшения**:
- 8.1.1.a. Для крипто заменить на: `1 + 0.15×sin(2π×hour/24) + 0.1×cos(2π×hour/24 - π/3)` — двойная гармоника для учёта утренней и вечерней активности
- 8.1.1.b. Убрать дневной множитель для крипто (7-дневная неделя), заменить на: `1 + 0.03×sin(2π×dayofweek/7)` — незначительная коррекция для выходных (пониженная активность)
- 8.1.1.c. Добавить **funding rate seasonality** для крипто-фьючерсов: каждые 8 часов (funding settlement) наблюдается повышенная волатильность → `1 + 0.2×exp(-((hour % 8 - 0)² / 0.5))` — гауссов пик вблизи funding time

### 8.2. Улучшение Confidence Scaling

#### 8.2.1. Проблема: При малых n прогноз приближен к 50%
**Текущее поведение**: `prob = 50 + (raw - 50) × sqrt(n/max_n)` — корректно для малых выборок.

**План улучшения**:
- 8.2.1.a. Добавить **Bootstrap-подтверждённый confidence**: для n ≥ 10, рассчитывать 90% CI через bootstrap (уже реализован в Monte Carlo) — если CI пересекает 50%, снижать confidence
- 8.2.1.b. Реализовать **Bayesian prior**: начинать с prior = 50% и обновлять через Bayes rule при каждом новом наблюдении — более плавная конвергенция чем текущий sqrt scaling
