## 📋 КОНТЕКСТ ПРОЕКТА

Система состоит из двух файлов:
- **Script_Latest_V1_Engine.txt** — `strategy()` (5579 строк): вычислительное ядро с Smart Money Concepts (SMC), зонами поддержки/сопротивления, Break Blocks, Rejection Zones, FVG, стохастической моделью, нейросетью, скорингом (6 компонент), управлением позициями и риском.
- **Script_Latest_V1_Visual.txt** — `indicator()` (443 строки): дашборд-визуализатор, получающий данные из ENGINE через 50+ `plot()` → `input.source()` каналов. Имеет локальные fallback-расчёты для автономной работы.

Связь между файлами: ENGINE экспортирует метрики через невидимые `plot()` (100% transparency), VISUAL импортирует их через `input.source()`. Переключатель `use_engine_data` определяет источник данных.

## 2. КРИТИЧЕСКИЕ БАГИ И ИСПРАВЛЕНИЯ

### 2.1. Незакреплённый результат (ENGINE, строка ~2805)
- 2.1.1. `math.min(confluence, 30.0)` — результат вычисления **не присваивается** переменной; должно быть `confluence := math.min(confluence, 30.0)` — **confluence score не ограничивается сверху**
- 2.1.2. Найти все аналогичные случаи через поиск паттерна `math.min(`, `math.max(` без `:=`

### 2.2. Инвертированная логика паттерна Harami (ENGINE, строки ~3322–3345)
- 2.2.1. Условие `body0_hi < body1_hi and body0_lo > body1_lo` проверяет что body0 ВНУТРИ body1 — но по определению Harami текущая свеча (body0) должна быть внутри предыдущей (body1) — **проверить правильность индексации `[0]` vs `[1]`**
- 2.2.2. Убедиться что переменные `body0_hi`, `body0_lo`, `body1_hi`, `body1_lo` соответствуют свече [0] (текущая) и [1] (предыдущая) соответственно
- 2.2.3. Написать unit-тест с конкретными значениями OHLC для верификации

### 2.3. Баг штрафа FVG-заполнения (ENGINE, строки ~4871–4875)
- 2.3.1. При `fvg_fill_pct > 0.5`: `score -= 10.0 * z.fvg_fill_pct` — при пороге 0.5 штраф = -5.0, при пороге 0.8 = -8.0 — **диапазон штрафа слишком узкий (5.0–8.0)**
- 2.3.2. Исправить на: `score -= 10.0 * (z.fvg_fill_pct - 0.5) / 0.3` для линейной интерполяции 0→33.3 в диапазоне [0.5, 0.8], или использовать `score -= 20.0 * z.fvg_fill_pct` для более агрессивного штрафа

### 2.4. Числовой underflow decay_factor (ENGINE, строки ~4856–4857)
- 2.4.1. `decay_factor = exp(-0.001 * age_bars)` — при age > 6900 значение < 0.001, при age > 10000 → 0.0
- 2.4.2. Добавить нижнюю границу: `decay_factor := math.max(0.01, math.exp(-0.001 * age_bars))` для предотвращения деления на ноль или невидимых зон

### 2.5. Буферная безопасность (ENGINE, строки 754–763)
- 2.5.1. Прямой доступ к `high[MAX_OFFSET]`, `low[MAX_OFFSET]` (строки 754–757) без проверки `bar_index >= MAX_OFFSET` — на первых барах (bar_index < 4998) возможен crash
- 2.5.2. Обернуть в условие: `if bar_index >= MAX_OFFSET` или использовать `nz(high[math.min(bar_index, MAX_OFFSET)], high)`
- 2.5.3. Аналогичная проблема в `f_analyze_pivot_quality()` при доступе к `volume[s_idx]`, `high[s_idx]` (строки 1799–1821) — `s_idx` ограничен через `f_clamp()`, но `max_bars_back()` вызван только для 3 серий (_mem_atr, gvs, rvol)

### 2.6. Дебаг-массив не инициализирован (ENGINE, строка ~1087)
- 2.6.1. `f_dbg(string s) => array.push(dbg_ch_log, s)` — `dbg_ch_log` не объявлен с `var` — **crash при первом вызове**
- 2.6.2. Добавить: `var array<string> dbg_ch_log = array.new<string>()`

### 2.7. Race condition при дублировании зон (ENGINE, строки ~4466–4471)
- 2.7.1. Линейный поиск по `zone_db` для проверки `origin_idx == last_sl_i` — при одновременном подтверждении нескольких пивотов возможно создание дубликатов
- 2.7.2. Добавить блокировку: `var int _last_created_idx = -1` и проверять `last_sl_i != _last_created_idx`

## 3. ЛОГИКА СКОРИНГА — АНАЛИЗ И УЛУЧШЕНИЕ

### 3.1. 6-компонентный скоринг (ENGINE, `f_compute_zone_score()`, строки ~3476–3503)
- 3.1.1. Текущие компоненты и веса по умолчанию:
  - Z-Score (25%) — сигмоидная нормализация аномальности тела свечи
  - Volume/RVOL (15%) — логарифмическая нормализация относительного объёма
  - Trend Alignment (15%) — бинарное совпадение с HTF/ITF трендом
  - Sweep/Liquidity Grab (15%) — наличие ликвидити-свипа (0.6 или 1.0 множитель)
  - Efficiency/FER (15%) — коэффициент эффективности движения цены
  - PA Bonus (15%) — Break + Engulfing + Sweep Vol паттерны
- 3.1.2. **Проблема**: Trend Alignment — бинарный (0 или full weight), без градации — **добавить промежуточные уровни** (partial alignment = 50% weight если ITF совпадает но HTF нет)
- 3.1.3. **Проблема**: Sweep компонент имеет 2 дискретных значения (0.6× или 1.0×) — **заменить на непрерывную функцию** от глубины свипа
- 3.1.4. Проверить что сумма весов нормализуется корректно (строки 93–100) — `_safe_sum = max(_sum_w, 0.001)` — при всех весах = 0 результат = 0.0, что корректно

### 3.3. Multi-pattern scoring (ENGINE, `f_multi_pattern_bonus()`, строки ~3431–3440)
- 3.3.1. Текущая таблица: 1 паттерн = +0%, 2 = +10%, 3 = +30%, 4 = +40%, 5+ = +50%
- 3.3.2. **Проблема**: скачок с +10% до +30% между 2 и 3 паттернами — **сгладить** логарифмической функцией
- 3.3.3. `MULTI_PATTERN_PENALTY = 0.85` (строка 26) — штраф если паттернов < минимума — **проверить** реальный эффект на backtesting

### 3.4. Скоринг Pivot Quality (ENGINE, `f_analyze_pivot_quality()`, строки ~1709–1955)
- 3.4.1. Четыре основных компонента: Curvature (s_c), Volume (s_v), Morphology (s_i), Fractal Depth (s_f)
- 3.4.2. Все 4 компонента ограничены [0, 100] — **но sweep_bonus не ограничен** (строка 1869: `sweep_bonus = min(p_sweep_liq_bonus, ...)` — зависит от пользовательского ввода)
- 3.4.3. Neighbor detection (строки 1874–1931): сканирование ±15 баров — **оптимизировать** через кэширование пивот-позиций вместо полного пересканирования
- 3.4.4. Логарифмическая нормализация curvature и fractal (`log(1+x)/log(4)`) — **обосновать** выбор основания 4 или заменить на адаптивный масштаб

### 3.5. Стохастическая модель (ENGINE, `f_get_stochastic_forecast()`, строки ~2498–2574)
- 3.5.1. Биварная модель (время + объём) с экспоненциальным затуханием весов: `w = exp(-2/n * age)`
- 3.5.2. Seasonality adjustment: `1 + 0.1*sin(2π*hour/24) + 0.05*sin(2π*day/5)` — **проблема**: криптовалюты и форекс не имеют 24-часовых циклов, активных дней может быть != 5
- 3.5.3. Confidence scaling: `prob = 50 + (raw - 50) * sqrt(n/max_n)` — при малых n (< 5) прогноз сильно приближен к 50% — **корректно**
- 3.5.4. CDF Hastings: точность 1.5e-7 — **достаточна**, оставить
- 3.5.5. **Улучшение**: добавить адаптивный lambda для exponential decay вместо фиксированного `2/n`

### 3.6. Нейросеть (ENGINE, строки ~2878–2924)
- 3.6.1. Архитектура: 6→4→1 с sigmoid activation
- 3.6.2. Online learning (Widrow-Hoff) с learning_rate = 0.01
- 3.6.3. **Проблема**: веса инициализируются `var` (сохраняются между барами), но при перезагрузке скрипта сбрасываются — **нет персистентности**
- 3.6.4. **Проблема**: 6 входов (z_score, rvol, trend, sweep, eff, pa) нормализованы вручную (`z_score/5`, `min(rvol,5)/5`) — **стандартизировать** через running mean/std
- 3.6.5. **Улучшение**: добавить регуляризацию (L2) для предотвращения переобучения на малых выборках
- 3.6.6. **Улучшение**: увеличить скрытый слой до 8–16 нейронов для лучшей экспрессивности

## 4. УПРАВЛЕНИЕ ЗОНАМИ — АНАЛИЗ И УЛУЧШЕНИЕ

### 4.1. Жизненный цикл зоны (Zone type, строки ~1958–2013)
- 4.1.1. Зона имеет 40+ полей — **разбить** на подтипы: `ZoneCore`, `ZoneVisual`, `ZoneScoring`, `ZoneFVG` для лучшей читаемости

  - `update_sniper()` — 3-стадийная рефайнмент логика
  - `update_fvg()` — FVG-сетка и удаление
  - `update_visuals()` — все графические элементы
  - `update_age()` — age fade и decay_factor

### 4.2. Breaker Block конвертация (ENGINE, `trigger_bb()`, строки ~2381–2477)
- 4.2.1. Пересчёт score при BB: `0.8 * old_score + momentum + bonuses` — **проблема**: cascade detection (bb_cascade_count ≥ 2) деактивирует зону, но визуальные элементы не удаляются немедленно — **добавить** `f_delete_zone_visuals()` в ветке cascade
- 4.2.2. BB cascade limit = 2 — **рассмотреть** параметризацию через input

### 4.3. Rejection Zone создание (ENGINE, строки ~4648–4802)
- 4.3.1. Sniper price = (top + bot) / 2 для RZ — **неоптимально**: точка входа должна быть ближе к wick rejection point, а не к midpoint
- 4.3.2. Adaptive threshold: `p_min_score × (0.4 + 0.2 × wick_ratio_norm)` — при большом wick_ratio порог снижается — **логично**, но множители (0.4, 0.2) — магические числа
- 4.3.3. Quadratic wick bonus: `pow(min(wick_ratio, 3), 2) × 20` — при wick_ratio = 3.0 bonus = 180 (!) — **превышает cap 150** — проверить что final capping работает корректно

### 4.4. Управление памятью зон
- 4.4.1. `var array<Zone> zone_db = array.new<Zone>()` — **неограниченный рост** — добавить жёсткий лимит (например, 500) с garbage collection старейших
- 4.4.2. `f_limit_visible_zones()` (строки 4276–4324) — ограничивает видимые зоны, но не удаляет невидимые из памяти — **memory leak**
- 4.4.3. Absorbed zones increment `absorbed_count` (строка 4215) но значение никогда не используется — **dead code**, удалить или реализовать

### 4.5. FVG Sequence Lock (ENGINE, строки ~4419–4430)
- 4.5.1. `active_bull_fvg_lock` блокирует создание новых бычьих зон если цена внутри любого активного бычьего FVG — **слишком агрессивно** при широких/старых FVG
- 4.5.2. **Улучшение**: ограничить lock только FVG моложе N баров или с fill_pct < 50%

## 5. СТРАТЕГИЧЕСКАЯ ЛОГИКА — АНАЛИЗ И УЛУЧШЕНИЕ

### 5.1. Вход в позицию (ENGINE, строки ~4945–5247)
- 5.1.1. Три режима входа: Touch (Limit), Candle Close, ChoCh (Confirm) — **оценить** win-rate каждого режима отдельно
- 5.1.2. Confluence filter: `min_confluence` (default 2) — проверяет наличие N+ факторов — **перечислить** все факторы: trend, VWAP, ADX, MSS, pattern, zone quality
- 5.1.3. Pin bar validation: проверка наличия pin bar на зоне для подтверждения — **добавить** проверку объёма на pin bar (high volume pin > low volume pin)

### 5.2. Риск-менеджмент (ENGINE, строки ~228–244, 5248–5274)
- 5.2.1. SL = zone boundary ± pad × ATR (default 0.5) — **корректно**
- 5.2.2. TP1 = 50% qty at fib_tp1 (1.618), TP2 = remaining at fib_tp2 (2.618) — **проверить** что partial fill обрабатывается корректно
- 5.2.3. Break-even trigger = 1.0R — **рассмотреть** динамический BE на основе волатильности (ATR-adjusted)
- 5.2.4. Trailing stop: активация 2.0R, offset 1.0R — **добавить** ATR-based trailing как альтернативу

### 5.3. Kelly Criterion (ENGINE, строки ~5336–5344)
- 5.3.1. `kelly_f = (win_rate × avg_win - (1-wr) × avg_loss) / avg_win` — **классическая формула**, корректна
- 5.3.2. Clamp [0, 1] — **слишком агрессивно**: рекомендуется Half-Kelly (`kelly_f / 2`) для снижения variance
- 5.3.3. Требуется минимум trades для стабильной оценки — **добавить** проверку `if total_trades >= 30`

### 5.4. Sharpe/Sortino Ratio (ENGINE, строки ~5358–5376)
- 5.4.1. Annualization: `× sqrt(252)` — **корректно для дневных данных**, но для внутридневных TF нужно `× sqrt(bars_per_year)` адаптивно
- 5.4.2. Minimum 5 trades — **увеличить** до 20 для статистической значимости

### 5.5. Monte Carlo Bootstrap (ENGINE, строки ~5378–5402)

### 12.3. Лимиты Pine Script
- 12.3.1. String limit 40960 chars — защита через `f_safe_append()` — **корректно**
